Git 使用笔记

Posted on 2018-05-20 | In Git

命令行版

git 为分布式版本管理系统

先进行Git全局设置

创建 git 仓库

如果项目存在，只需在添加远程仓库开始即可。

通过 git init 将目录变成 git 管理的目录，此时目录下会出现 .git 文件夹，里面会存储版本管理文件

文件格式：需要使用纯文本编辑的文件才能追踪变化

步骤：

  1. 先创建文件
  2. 通过 git add <file>将文件添加到仓库
  3. 通过 git commit -m "<说明>"将文件提交到仓库

解释 git commit 参数：

-m：后面输入的是本次提交的说明

git status: 查看当前仓库状态

git diff: 查看difference

版本回退：

git log: 显示从最近到最远的提交日志

git reset --hard HEAD^: 回退到上个commit版本，其中HEAD^表示上个版本，如果想找回最新的版本，只能通过之前保留的commitId（只需前几位）进行恢复

工作区和暂存区：

工作区：电脑中能看到的目录

版本库：.git文件夹，里面存有暂存区，需要提交的文件通通放到暂存区，然后一次性提交暂存区的文件，每次修改需要add到暂存区后再commit

撤销修改：

git checkout -- <file>: 使用指令丢掉缓冲区的修改，也可以丢掉工作区的修改

这里会存在两种情况：

  1. 一种是文件还没有被add到暂存区，现在撤销修改就会回到和版本库一样的状态
  2. 一种是文件已经被添加到暂存区后，又做了修改，撤销修改就会回到add到暂存区的状态

git reset HEAD <file>: 该命令既可以回退版本，也可以把暂存区的修改撤销掉，重新放回工作区

删除文件：

假如有一个文件被add而且commit后，在本地删除了该文件。此时工作区和版本库不一样，通过git status查看后可以发现文件被删除

  1. 如果选择删除文件，可以使用git rm <file>后commit
  2. 如果是误删文件，可使用git checkout --<file>恢复，因为该命令实际上是用版本库的版本替换工作区的版本，无论是工作区删除还是修改。

添加远程仓库：

本地git和github仓库之间的传输是通过ssh加密的

第一次将本地仓库推送到远程，需要先将本地仓库与远程仓库关联git remote add origin git@<username>/<categories>.git

使用git push -u origin master，origin为远程仓库名，git会把本地的master分支内容推送到远程的新的master分支，还会把远程的master分支和本地的master关联起来，以后修改只需通过git push origin master就可以将内容推送到服务器上

从远程仓库克隆：

先有远程仓库的话直接使用git clone命令克隆，或者使用ssh链接克隆策略

分支管理：

git checkout -b <name>: 创建分支名，其中-b表示新建并切换

git branch: 查看当前分支

git merge <branch>: 合并指定分支到当前分支

git branch -d <branch>: 删除指定分支

解决合并冲突：

当git无法自动合并分支的时候，首先要解决冲突，再提交，合并完成。

还可以使用git -log --graph查看分支合并图

分支管理策略：

使用--no-ff禁用fast forword，Git会在merge的时候生成一个新的commit，这样在分支历史中就可以看到分支信息。

一般在dev分支干活，master分支是非常稳定的，然后每个人在dev分支上再分自己的工作流

Bug分支：

git stash: 在开发中，假如遇到需要紧急修复bug的时候，此时没有办法提交的时候，可以使用此命令将工作现场存储起来，等以后恢复现场的时候再继续工作

在时候git stash后使用git status查看工作区可以发现此时工作区是空的，此时就可以切换回出现bug的分支，在此基础上新建分支，修复完成后合并删除分支，修复完成后切换回dev分支，使用git stash list查看工作现场列表

git stash apply: 恢复工作现场，保留stash文件

git stash pop: 恢复工作分区，删除stash文件

Feature分支：

和Bug分支类似，没添加一个新功能，最好新建一个feature分支，然后在上面开发，完成后，合并并删除该分支

git branch -D <name>: 强行删除分支

推送分支：

git remote -v: 查看远程仓库详细信息

git push origin <branch>: 将该分支上的所有本地提交推送到远程库，推送的时候要指定本地分支

一般来说master和dev是要和远程保持一致，而其他分支则看情况

抓取分支：

  1. 首先，可以试图用git push origin branch-name推送自己的修改；
  2. 如果推送失败，则因为远程分支比你的本地更新，需要先用git pull试图合并；
  3. 如果合并有冲突，则解决冲突，并在本地提交；
  4. 没有冲突或者解决掉冲突后，再用git push origin branch-name推送就能成功！

如果git pull提示“no tracking information”，则说明本地分支和远程分支的链接关系没有创建，用命令git branch --set-upstream branch-name origin/branch-name。

这就是多人协作的工作模式，一旦熟悉了，就非常简单。
